# GovCMS 发布中心 MVP PRD

## 1. 文档信息

- 文档名称：发布中心 MVP PRD
- 版本：V1.0
- 所属阶段：`L1 CMS 平台共享层` / `L2 后台标准版`
- 当前状态：待评审
- 最近更新：2026-03-16
- 关联文档：`docs/09-content-lifecycle-prd.md`、`docs/10-publish-render-contract-prd.md`、`docs/14-publish-render-technical-design.md`
- 适用范围：站点、栏目、内容、模板等对象的正式发布、下线、重试、回滚

## 2. 背景与定位

当前项目已经具备栏目管理、模板管理和受控预览能力，但还没有一个统一的“正式发布入口”来决定：

- 哪些对象可以正式发布
- 影响范围如何计算
- 发布结果如何确认
- 失败如何处理
- 回滚如何执行与留痕

发布中心 MVP 的目标不是一步做成完整 DevOps 发布平台，而是先建立从 CMS 对象到门户正式产物之间的最短业务闭环。

## 3. 本期目标

### 3.1 目标

- 建立统一发布入口，避免各模块各自发布
- 支持站点、栏目、内容、模板四类发布单位
- 在执行前给出可理解的影响范围
- 形成可追踪的发布任务、执行日志、产物与回滚记录
- 为后续门户正式静态化和增量发布打基础

### 3.2 非目标

本期不包含以下能力：

- 分布式多节点并行发布
- 蓝绿发布、灰度发布
- CDN 自动刷新编排
- 多环境差异编排
- 多租户跨站点联动发布
- 全自动无人值守发布

## 4. 与当前项目进度的关系

截至 2026-03-16，当前代码已经具备以下前提：

- 栏目对象与栏目影响范围基础
- 模板对象、模板绑定、模板版本与模板影响范围基础
- `RenderContextAssembler` 与 `PortalRenderService` 的受控预览基础
- 内容基础 CRUD 与发布 / 下线按钮雏形

当前仍缺少：

- 正式发布任务模型
- 统一影响范围计算入口
- 正式发布成功与失败的结果记录
- 回滚入口与回滚记录

因此，本 PRD 聚焦定义“发布中心 MVP”，而不是一次性定义完整渲染引擎细节。

## 5. 核心产品决策

### 5.1 发布单位固定为四类

MVP 首期支持以下四类发布单位：

- `content`：内容
- `category`：栏目
- `template`：模板
- `site`：站点

说明：

- `content` 用于详情页、栏目聚合位的增量变更
- `category` 用于栏目页、导航、面包屑等结构性变更
- `template` 用于模板变更导致的页面重建
- `site` 用于整站发布或站点级关键配置变更

### 5.2 模板变更自动计算影响范围，但不自动执行正式发布

模板变更后，系统应自动计算影响范围并提示可能受影响的页面与对象，但默认不自动执行正式发布。

原因：

- 模板变更影响面广，直接自动发布风险过高
- 需要人工确认发布窗口与影响面
- 这样既保留 CMS 专业能力，又避免误发版

### 5.3 预览与正式发布严格分离

- 预览：用于内部验证，可使用草稿、样例数据、最新模板，不形成正式产物，不进入发布审计主链路
- 正式发布：只能基于已批准对象和锁定版本快照执行，必须生成任务记录、日志、产物清单和可回滚记录

### 5.4 回滚按任务发起，按产物集执行

从产品视角，回滚对象是一次发布任务；从实现视角，回滚恢复的是该任务涉及的稳定产物集合。

这样最符合用户理解，也方便审计留痕。

## 6. 适用角色

| 角色 | 使用目标 |
| --- | --- |
| 发布员 | 发起发布、执行发布、查看结果、重试失败、执行回滚 |
| 站点管理员 | 查看站点级发布情况、执行整站发布、处理异常 |
| 审核员 | 查看已批准对象是否进入发布链路 |
| 平台管理员 | 查看全局日志、审计、异常任务处理 |

## 7. MVP 功能范围

### 7.1 必做能力

- 发布前校验
- 影响范围计算
- 创建发布任务
- 执行发布任务
- 查看任务列表与详情
- 查看执行日志
- 查看产物清单
- 重试失败任务
- 执行任务级回滚

### 7.2 明确不做

- 定时发布中心
- 审批后自动发布策略配置
- 蓝绿 / 灰度 / 多环境编排
- 搜索索引复杂异步编排
- CDN 自动刷新与外部系统联动

## 8. 发布单位与影响规则

### 8.1 内容发布

典型触发：

- 内容从 `approved` 进入正式发布
- 已发布内容执行下线

典型影响：

- 内容详情页
- 所属栏目列表页
- 首页 / 专题页中的聚合位（若有）
- 后续可扩展为搜索索引

### 8.2 栏目发布

典型触发：

- 栏目名称、路径、排序、父子结构变化
- 栏目模板配置变化

典型影响：

- 栏目列表页
- 导航
- 面包屑
- 子栏目关联路径
- 受栏目模板影响的详情页聚合关系

### 8.3 模板发布

典型触发：

- 模板新版本生效
- 模板绑定变更
- 模板解绑

典型影响：

- 所有绑定该模板的页面集合
- 栏目页 / 详情页 / 首页等对应渲染页面

### 8.4 站点发布

典型触发：

- 站点关键配置变化
- 无法准确确定影响范围时的兜底整站发布
- 首次全量生成门户站点

典型影响：

- 整站页面与产物集合

## 9. 核心业务流程

### 9.1 已批准内容正式发布

1. 发布员选择状态为 `approved` 的内容
2. 系统执行发布前校验
3. 系统计算影响范围
4. 发布员确认后创建发布任务
5. 执行发布任务
6. 成功后内容状态写回 `published`
7. 记录产物与日志

### 9.2 模板变更后的正式发布

1. 模板发生新版本或绑定变化
2. 系统自动给出影响范围预估
3. 发布员确认发布窗口与影响面
4. 创建模板类发布任务
5. 执行成功后记录影响对象与产物更新

### 9.3 站点整站发布

1. 发布员选择站点发布
2. 系统说明这是全量发布
3. 创建整站发布任务
4. 执行并记录产物、日志和结果摘要

### 9.4 发布失败重试

1. 任务执行失败
2. 发布员查看失败原因和失败节点
3. 对可重试任务执行重试
4. 重试形成新的执行记录，不覆盖原始失败记录

### 9.5 回滚

1. 发布员从稳定成功任务发起回滚
2. 系统展示目标任务、目标版本、受影响范围
3. 确认后执行回滚任务
4. 回滚完成后形成新的任务记录与日志

## 10. 发布状态模型

### 10.1 任务状态建议

MVP 首期建议使用以下状态：

- `created`：已创建，待执行
- `checking`：校验中
- `ready`：校验通过，待确认 / 待执行
- `running`：执行中
- `success`：执行成功
- `failed`：执行失败
- `rollback_success`：回滚成功
- `rollback_failed`：回滚失败

### 10.2 状态说明

- `created` 与 `ready` 区分“任务已建立”和“校验已通过”
- `success` 才能作为稳定版本来源
- 回滚应形成新的任务记录，而不是覆盖旧任务状态

## 11. 页面设计建议

### 11.1 发布中心首页

建议包含以下区域：

- 发布入口区：整站发布 / 增量发布
- 待发布对象区：可选择内容、栏目、模板、站点
- 影响范围预览区
- 待执行任务区
- 历史任务列表区

### 11.2 任务详情页

建议包含以下区域：

- 基础信息区：任务类型、站点、操作者、时间、状态
- 触发对象区
- 影响范围区
- 执行日志区
- 产物清单区
- 重试 / 回滚操作区

### 11.3 任务列表筛选项

至少支持：

- 站点
- 任务类型
- 发布单位类型
- 任务状态
- 时间范围
- 操作人

## 12. 前置校验规则

### 12.1 通用校验

- 站点有效
- 目标对象存在
- 当前用户具备发布权限
- 渲染上下文可组装
- 影响范围可计算

### 12.2 内容发布校验

- 内容状态必须为 `approved`
- 所属栏目有效
- 详情模板可定位
- 关键媒体可访问

### 12.3 下线校验

- 内容当前状态必须为 `published`
- 下线影响范围可识别

### 12.4 模板发布校验

- 模板状态有效
- 模板版本明确
- 模板绑定关系完整

### 12.5 降级策略

- 若影响范围无法准确识别，系统必须提示改走整站发布
- 不允许在影响范围不清晰的情况下直接执行不安全增量发布

## 14. 权限建议

建议发布中心 MVP 使用以下权限编码：

| 权限编码 | 说明 |
| --- | --- |
| `publish:view` | 查看任务列表和详情 |
| `publish:impact:view` | 查看影响范围 |
| `publish:execute` | 执行发布 |
| `publish:retry` | 重试失败任务 |
| `publish:rollback` | 执行回滚 |
| `publish:artifact:view` | 查看产物清单 |
| `publish:log:view` | 查看日志 |

## 15. 接口建议

建议首期接口如下：

- `POST /api/publish/precheck`：发布前校验
- `POST /api/publish/impact`：计算影响范围
- `POST /api/publish/jobs`：创建发布任务
- `POST /api/publish/jobs/{id}/execute`：执行发布任务
- `POST /api/publish/jobs/{id}/retry`：重试失败任务
- `POST /api/publish/jobs/{id}/rollback`：发起回滚
- `GET /api/publish/jobs`：任务列表
- `GET /api/publish/jobs/{id}`：任务详情
- `GET /api/publish/jobs/{id}/artifacts`：产物清单
- `GET /api/publish/jobs/{id}/logs`：任务日志

## 16. 首期验收标准

### 16.1 功能验收

- 可对已批准内容发起正式发布
- 可对栏目、模板、站点发起发布任务
- 可在执行前查看影响范围
- 可查看任务详情、日志和产物清单
- 失败任务可重试
- 成功任务可回滚

### 16.2 业务验收

- 内容审核通过后不会自动对门户可见，必须经过发布中心
- 模板变更可自动给出影响范围，但不会自动正式发布
- 影响范围无法明确时系统能提示走整站发布
- 回滚后门户恢复到目标稳定版本

### 16.3 风险控制验收

- 任务失败时不会错误标记内容为已发布
- 门户不会消费未发布对象
- 回滚过程完整留痕
- 发布日志、产物、任务状态相互一致

## 17. 后续扩展位

以下能力保留到后续阶段：

- 定时发布
- 审批后自动发版策略
- 搜索索引单独编排
- CDN 刷新联动
- 多节点并行渲染
- 蓝绿 / 灰度 / 多环境发布

## 18. 本期结论

发布中心 MVP PRD v1 的目标不是做“大而全”的发布平台，而是先建立一条最短、可审计、可回滚的正式发布主链路：

- 以内容生命周期的 `approved` 作为发布输入
- 以影响范围计算作为发布决策前置
- 以发布任务作为正式发布唯一入口
- 以任务级回滚保证门户产物可恢复

对于当前项目进度而言，这份 PRD 将成为栏目、模板、内容生命周期和门户发布之间的产品桥梁，也是后续审核中心、发布中心和门户标准版继续推进的最优起点。
