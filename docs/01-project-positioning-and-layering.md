# GovCMS 项目定位与分层说明

## 1. 项目定位

GovCMS 不是单一的后台管理页面集合，而是一个面向政务场景的 CMS 平台项目，目标交付包括：

- CMS 平台后端源码
- 管理后台源码
- 基于 CMS 建站并发布的门户站点能力
- 静态化发布体系
- 完整数据库脚本、部署脚本、配置模板、运维文档、验收文档
- 信创适配能力
- 国密安全能力

门户网站的统一定义如下：

- 门户由 CMS 后台驱动建站，不是独立维护的另一套内容系统。
- 运营人员在 CMS 后台完成站点、栏目、内容、专题、导航、模板配置。
- 发布后生成门户静态页面和静态资源，对外以静态站点为主运行。
- 搜索、表单提交、统计上报等少量能力允许保留动态接口支持。

## 2. 项目分层

| 层级 | 名称 | 说明 | 当前状态 |
| --- | --- | --- | --- |
| L0 | 当前基座 | 当前仓库已实现的后台 MVP 原型能力 | 已交付 |
| L1 | CMS 平台共享层 | 站点、栏目、内容、媒体、专题、模板、发布、搜索索引、安全、审计等共享能力 | 规划中 |
| L2 | 后台标准版 | 面向站点管理员、编辑、审核人员的完整后台工作台 | 规划中 |
| L3 | 门户标准版 | 首页、栏目页、详情页、专题页、搜索与静态化发布门户 | 规划中 |
| L4 | 扩展业务层 | 正能量稿池、举报辟谣、多租户、复杂工作流等扩展能力 | 储备 |

## 3. 当前代码归位

当前仓库代码只应被描述为 `L0 当前基座`，不能再用“已完成整站方案”“已完成信创交付”“已完成正式门户建设”等表述。

`L0` 已交付范围：

- 登录认证与 JWT 鉴权
- RBAC 权限模型
- 动态菜单和按钮级权限控制
- 仪表盘
- 用户、角色、权限、菜单管理
- 内容基础管理（文章 CRUD、发布、下线）
- 站点基础管理（列表、筛选、新增、编辑、删除、启停）
- 媒体基础管理（上传、列表、筛选、预览、删除）
- 本地文件存储实现
- 本地和测试环境配置

`L0` 尚未覆盖范围：

- 栏目树与栏目模板
- 专题页与页面模板
- 发布中心与静态化门户发布
- 正式审核流
- 审计日志与安全运维视图
- KingbaseES、TongWeb、国产 OS、政务云部署方案落地
- 国密传输、国密签名、国密数据保护落地

## 4. 统一状态标签

今后所有文档、汇报和验收清单都只能使用以下状态标签：

- `已交付`：代码、文档、验证已完成，可演示或可交付。
- `设计完成`：设计口径与接口边界已定稿，但代码未完成。
- `开发中`：已经进入实现阶段，但尚未完成交付。
- `规划中`：还在方案或排期层面。
- `储备`：明确有价值，但不进入当前阶段承诺。

## 5. 描述口径约束

必须这样描述：

- 当前代码是“后台 MVP 原型基座”。
- 首期正式目标是“CMS 平台共享层 + 后台标准版 + 门户标准版”。
- 信创适配与国密要求是正式交付基线。

不能这样描述：

- “当前项目已经完成门户建设”
- “当前项目已经完成信创部署”
- “当前项目已经完成等保三级落地”
- “当前项目已经完成全量方案”

## 6. 交付口径

正式交付不等于只交付门户静态页，而是交付整套工程资产：

- 后端源码
- 后台前端源码
- 门户模板与发布相关源码
- 数据库脚本
- 部署脚本
- 环境配置模板
- 运维与回滚文档
- 测试与验收文档

门户“静态化”描述的是门户对外运行方式，不是交付范围的缩减。
