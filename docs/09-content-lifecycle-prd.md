# GovCMS 内容生命周期 PRD

## 1. 文档信息

- 文档名称：内容生命周期 PRD
- 版本：V1.0
- 所属阶段：`L1 CMS 平台共享层` / `L2 后台标准版`
- 当前状态：待评审
- 最近更新：2026-03-16
- 关联文档：`docs/07-column-management-prd.md`、`docs/08-template-management-prd.md`、`docs/10-publish-render-contract-prd.md`、`docs/15-publish-center-mvp-prd.md`
- 适用范围：CMS 内容创建、编辑、审核、发布、下线、回滚协同

## 2. 背景与定位

当前仓库已经具备栏目管理、模板管理和受控预览能力，但“内容如何从草稿流转到门户正式可见”仍缺少统一产品规则。

内容生命周期模块是 GovCMS 从“后台可编辑”走向“门户可交付”的第一条业务主链路。它负责解决以下问题：

- 内容有哪些正式状态
- 每个状态谁能操作、能做什么
- 审核与发布的边界如何区分
- 门户什么时候可以消费内容
- 内容下线、重新送审、回滚后如何保持一致

它不是简单的状态字段，而是一套覆盖内容、栏目、模板、审核、发布、门户读取边界的统一规则。

## 3. 本期目标

### 3.1 目标

- 建立清晰、稳定、可审计的内容状态模型
- 建立编辑、审核、发布三类职责分离机制
- 保证门户只消费正式发布成功的内容
- 为发布中心 MVP 提供稳定触发条件
- 为后续多级审核、定时发布、复杂工作流预留扩展位

### 3.2 非目标

本期不包含以下能力：

- 多级审批、会签、转签、加签
- 自定义状态机设计器
- AI 自动审核和质量评分
- 复杂排期中心
- 跨站点联动审批

## 4. 与当前项目进度的关系

截至 2026-03-16，项目已经具备以下基础：

- 内容基础 CRUD
- 栏目管理与栏目归属能力基础
- 模板绑定与预览能力基础
- 受控 HTML 预览能力

当前缺口在于：

- 没有统一内容状态机
- 没有审核角色与发布角色边界
- 没有“审核通过”和“正式发布完成”的明确区分
- 没有与发布任务的稳定协同规则

因此，本 PRD 的目标不是重做内容页面，而是为后续“审核中心 + 发布中心 + 门户发布”建立共同业务规则。

## 5. 核心产品决策

### 5.1 首期固定六态模型

首期内容状态固定为六态：

- `draft`：草稿
- `pending_review`：待审核
- `rejected`：已驳回
- `approved`：审核通过，待发布
- `published`：已发布
- `offline`：已下线

### 5.2 审核通过与正式发布分离

`approved` 与 `published` 必须分离，不能合并。

原因：

- 审核通过只代表业务内容达标，不代表门户已经完成更新
- 后续会存在“审核通过但发布失败”的情况
- 发布中心必须以“可发布对象”作为输入，而不是直接消费审核动作

### 5.3 生命周期状态与发布任务状态分离

内容状态用于表达内容业务可见性；发布任务状态用于表达发布执行过程。

- 内容状态不承载 `running`、`failed`、`retrying` 等执行语义
- 发布任务状态不替代内容状态

### 5.4 角色职责分离

首期按以下角色模型设计：

- 编辑：创建、编辑、提审、查看驳回意见
- 审核员：通过、驳回
- 发布员：执行发布、下线、回滚
- 站点管理员：配置、兜底处理异常、查看全链路记录

允许同一账号拥有多个角色，但产品语义必须分离。

## 6. 适用角色

| 角色 | 使用目标 |
| --- | --- |
| 平台管理员 | 管理全局规则、权限、审计查看 |
| 站点管理员 | 站点级异常处理、内容分布查看、兜底发布处理 |
| 编辑 | 创建内容、编辑内容、送审、查看驳回原因、再次送审 |
| 审核员 | 审核通过、驳回、查看审核上下文 |
| 发布员 | 执行发布、下线、回滚、查看发布结果 |

## 7. 业务边界

生命周期模块与其他模块的关系如下：

- 与 `Content`：生命周期直接作用于内容主对象
- 与 `Category`：内容必须归属有效栏目才能进入审核 / 发布链路
- 与 `Template`：内容发布前必须能定位到可用详情模板
- 与 `MediaAsset`：关键媒体引用必须有效
- 与 `PublishJob`：内容状态从 `approved` 进入发布任务，由任务成功后写回 `published`
- 与 `Portal`：门户只消费 `published` 状态内容
- 与 `AuditLog`：关键操作必须留痕

## 8. 生命周期模型

### 8.1 状态定义

| 状态 | 中文名 | 含义 | 门户可见 |
| --- | --- | --- | --- |
| `draft` | 草稿 | 编辑中的未提交内容 | 否 |
| `pending_review` | 待审核 | 已提交审核，等待审核结果 | 否 |
| `rejected` | 已驳回 | 审核未通过，等待修订 | 否 |
| `approved` | 审核通过待发布 | 已审核通过，但尚未完成正式发布 | 否 |
| `published` | 已发布 | 已完成正式发布，门户可消费 | 是 |
| `offline` | 已下线 | 曾发布但已下线，不再门户展示 | 否 |

### 8.2 标准流转

首期只允许以下标准流转：

- `draft -> pending_review`
- `pending_review -> rejected`
- `pending_review -> approved`
- `rejected -> draft`
- `rejected -> pending_review`
- `approved -> published`
- `published -> offline`
- `offline -> pending_review`

### 8.3 首期禁止流转

首期不允许以下跳转：

- `draft -> approved`
- `draft -> published`
- `pending_review -> draft`
- `approved -> draft`
- `offline -> published`
- `rejected -> published`

### 8.4 特殊说明

- 内容从 `approved` 到 `published` 必须经过发布中心执行成功
- `offline` 后若要再次上线，必须重新进入 `pending_review`
- 驳回后允许直接再次送审，也允许先回到 `draft` 修订后再送审

## 9. 角色与动作矩阵

| 状态 | 编辑 | 审核员 | 发布员 | 站点管理员 |
| --- | --- | --- | --- | --- |
| `draft` | 编辑、删除、提审、预览 | 只读 | 只读 | 可读、可协助 |
| `pending_review` | 只读、撤回（可选） | 通过、驳回 | 只读 | 可读 |
| `rejected` | 查看原因、修订、提审 | 只读 | 只读 | 可读 |
| `approved` | 只读 | 只读 | 发布、查看影响范围 | 可发布 |
| `published` | 只读 | 只读 | 下线、发起回滚 | 可下线、回滚 |
| `offline` | 只读 | 只读 | 发起复审、查看记录 | 可发起复审 |

说明：

- 首期建议不允许编辑在 `pending_review` 期间继续修改正文，避免“审核看到的版本”和“最终发布版本”不一致
- 若需要“撤回提审”，建议作为可选能力进入二期，不纳入本期强承诺

## 10. 核心业务流程

### 10.1 新建与保存草稿

1. 编辑创建内容
2. 默认进入 `draft`
3. 可多次保存草稿与预览
4. 草稿不可对门户可见

### 10.2 提交审核

1. 编辑点击“提交审核”
2. 系统执行基础校验：标题、栏目、正文、模板可用性、关键媒体有效性
3. 校验通过后进入 `pending_review`
4. 记录送审时间、送审人、版本号

### 10.3 审核通过

1. 审核员查看内容和预览结果
2. 点击“审核通过”
3. 内容进入 `approved`
4. 系统提示“可进入发布中心执行正式发布”

### 10.4 审核驳回

1. 审核员填写驳回原因
2. 内容进入 `rejected`
3. 编辑可查看原因并修订
4. 驳回信息必须保留历史记录

### 10.5 正式发布

1. 发布员在发布中心选择已 `approved` 的内容
2. 系统计算影响范围并执行发布任务
3. 任务成功后，内容状态写回 `published`
4. 若任务失败，内容保持 `approved`，不得误写为 `published`

### 10.6 下线

1. 发布员对 `published` 内容执行下线
2. 系统生成下线类发布任务
3. 任务成功后，内容进入 `offline`
4. 门户不再展示对应内容

### 10.7 再次上线

1. `offline` 内容若需重新上线
2. 必须重新进入 `pending_review`
3. 审核通过后重新进入 `approved`
4. 再由发布中心执行发布，成功后回到 `published`

## 11. 发布前置校验

内容进入 `pending_review` 或进入发布中心前，应至少校验：

- 标题非空
- 所属站点有效
- 所属栏目有效且状态可用
- 内容正文非空或具备有效结构化内容
- 详情模板可定位且状态有效
- 关键媒体引用可访问
- 内容无明显缺失字段

发布中心额外校验：

- 内容状态必须为 `approved` 或允许下线的 `published`
- 关联栏目、模板、站点未被禁用
- 渲染上下文可组装
- 影响范围可计算

## 12. 页面与交互要求

### 12.1 内容列表页

至少展示：

- 标题
- 所属站点
- 所属栏目
- 当前状态
- 最近操作时间
- 最近操作人

### 12.2 内容详情 / 编辑页

至少展示：

- 基础信息区
- 正文编辑区
- 栏目与模板关联信息区
- 生命周期状态区
- 审核意见区
- 操作记录区

### 12.3 审核工作区

至少支持：

- 按站点 / 栏目 / 状态筛选待审核内容
- 查看内容详情与预览
- 填写驳回原因
- 执行通过 / 驳回

## 15. 首期验收标准

### 15.1 功能验收

- 编辑可创建草稿并提交审核
- 审核员可通过或驳回内容
- 驳回原因可追溯
- 审核通过后内容进入 `approved`，而不是直接对门户可见
- 发布成功后内容才进入 `published`
- 下线成功后内容进入 `offline`

### 15.2 业务验收

- 门户不会消费 `draft`、`pending_review`、`rejected`、`approved`、`offline` 内容
- 审核通过但发布失败时，内容不会误展示到门户
- 已下线内容重新上线必须重新审核
- 审核、发布、下线、回滚记录可追溯

### 15.3 风险控制验收

- 审核中内容不可被隐式修改并直接发布
- 无权限角色不能越权通过审核或发布
- 状态流转非法时返回明确错误
- 生命周期日志与发布任务日志可互相关联

## 16. 后续扩展位

以下能力保留到后续阶段：

- 撤回提审
- 多级审批
- 定时发布
- 审核 SLA 与催办
- 敏感词 / AI 审核
- 内容冻结与并发编辑冲突控制

## 17. 本期结论

内容生命周期 PRD v1 的目标，不是把所有工作流一步到位，而是先把“编辑、审核、发布”三条职责边界固定下来。

对于当前项目进度而言，最重要的是把 `approved` 和 `published` 分开，把生命周期状态和发布任务状态分开，并让发布中心成为正式发布的唯一入口。这样才能在已完成的栏目、模板、受控预览基础上，继续向审核中心、发布中心和门户发布闭环推进。
