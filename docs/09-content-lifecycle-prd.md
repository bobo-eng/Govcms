# GovCMS 内容生命周期 PRD

## 1. 文档信息

- 文档名称：内容生命周期 PRD
- 所属阶段：`L1 CMS 平台共享层` / `L2 后台标准版`
- 当前状态：V1.0 初稿
- 最近更新：2026-03-11
- 适用范围：CMS 内容创建、编辑、审核、发布、下线、回溯、静态化发布协同

## 2. 产品定位

内容生命周期是 GovCMS 中连接“内容生产”“内容审核”“内容发布”和“门户生成”的核心规则模块。

它不是单纯的状态字段设计，而是一套约束内容在系统内如何流转、谁可以操作、何时可发布、何时可下线、何时触发门户更新的统一规则。

在 GovCMS 的整体体系中，内容生命周期承担以下职责：

- 统一内容状态定义
- 统一角色操作边界
- 统一审核与发布前置条件
- 统一下线与回退规则
- 为发布中心提供稳定的触发条件
- 为门户展示提供“只消费已发布内容”的明确边界

## 3. 产品目标

### 3.1 目标

- 建立清晰、稳定、可追溯的内容状态流转规则
- 支持编辑、审核、运维之间的职责分离
- 保证门户仅消费“已发布内容”，不暴露草稿和审核中内容
- 支持发布、下线、驳回、重新送审等首期核心动作
- 为后续多级审核、定时发布、复杂工作流预留扩展位

### 3.2 非目标

首期内容生命周期不负责以下能力：

- 多级审批流编排
- 任意自定义状态机
- 条件审批、会签、加签、转签
- AI 自动审核与内容质量评分
- 复杂定时排程中心

## 4. 适用角色

| 角色 | 使用目标 |
| --- | --- |
| 平台管理员 | 管理全局状态规则、权限边界、审计查看 |
| 站点管理员 | 查看站点内容状态分布、处理异常发布情况 |
| 编辑 | 创建草稿、编辑内容、送审、修订、重新送审 |
| 审核人员 | 审核内容、驳回内容、确认发布前状态 |
| 运维人员 | 执行发布、下线、回滚和发布异常处理 |

## 5. 业务边界

内容生命周期与其他模块的关系如下：

- 与 `Content`：生命周期直接作用于内容主对象
- 与 `Column`：内容发布前必须归属有效栏目
- 与 `Template`：内容发布前必须能定位到有效详情模板
- 与 `MediaAsset`：内容引用的关键媒体必须可访问且状态有效
- 与 `PublishJob`：内容状态进入 `published` 或 `offline` 时触发发布影响计算
- 与 `Portal`：门户仅消费 `published` 状态内容
- 与 `AuditLog`：所有关键状态流转必须产生审计记录

## 6. 生命周期模型

### 6.1 首期状态定义

首期固定采用以下状态模型：

- `draft`：草稿
- `review`：审核中
- `rejected`：已驳回
- `published`：已发布
- `offline`：已下线

### 6.2 状态含义

| 状态 | 含义 | 门户可见 |
| --- | --- | --- |
| `draft` | 编辑中的未提交内容 | 否 |
| `review` | 已送审，等待审核结果 | 否 |
| `rejected` | 审核未通过，需编辑修订 | 否 |
| `published` | 已通过审核并完成正式发布 | 是 |
| `offline` | 曾发布但已下线，不再对外展示 | 否 |

### 6.3 首期状态流转

首期只允许以下标准流转：

- `draft -> review`
- `review -> rejected`
- `review -> published`
- `rejected -> draft`
- `rejected -> review`
- `published -> offline`
- `offline -> review`

首期不允许：

- `draft -> published`
- `draft -> offline`
- `review -> draft`
- `offline -> published` 直接跳转

## 7. 核心场景

### 7.1 编辑创建内容

编辑创建内容后，默认进入 `draft`，可多次保存与预览，但不能直接对外发布。

### 7.2 送审

编辑确认内容基本信息、栏目归属、模板渲染、媒体引用均满足要求后，提交审核，状态进入 `review`。

### 7.3 驳回

审核人员发现内容存在问题时，将内容驳回并填写驳回原因，状态进入 `rejected`。

### 7.4 重新编辑与再送审

编辑查看驳回原因后修订内容，可回到 `draft` 继续编辑，完成后再次送审。

### 7.5 发布

审核通过后，由发布中心或具备权限的操作人执行发布，状态进入 `published`，门户详情页和相关聚合页可对外展示。

### 7.6 下线

当内容过期、错误或不再适合展示时，可执行下线，状态进入 `offline`，门户不再展示该内容。

### 7.7 重新启用已下线内容

已下线内容如果需要再次上线，必须重新进入 `review`，重新审核通过后才能再次发布。

## 8. 核心设计原则

- 门户只消费已发布内容，绝不消费草稿、审核中或驳回内容
- 编辑和审核职责分离，避免自审自发
- 发布和下线必须是显式动作，不能由编辑保存自动触发
- 所有关键状态变更必须可审计、可回溯
- 生命周期必须与栏目、模板、发布契约联动，而不是孤立存在

## 9. 功能范围

### 9.1 状态展示

内容列表、详情页和工作台都需要清晰展示当前生命周期状态。

功能要求：

- 列表显示状态标签
- 显示最后更新时间和最后操作人
- 对审核中、已驳回、已发布、已下线提供清晰区分
- 支持按状态筛选内容

### 9.2 草稿管理

编辑在 `draft` 状态下可执行以下操作：

- 保存草稿
- 继续编辑
- 预览内容
- 删除草稿
- 提交审核

首期规则：

- 草稿保存不限次数
- 草稿未送审前不可进入发布中心

### 9.3 送审

送审是 `draft` 或 `rejected` 进入 `review` 的标准入口。

送审前校验至少包括：

- 标题非空
- 正文非空
- 已归属站点与栏目
- 栏目状态有效
- 详情模板可解析
- 关键媒体引用有效

若校验失败，系统必须明确提示失败原因。

### 9.4 审核

审核人员在 `review` 状态下可执行以下操作：

- 通过审核
- 驳回审核
- 查看预览
- 查看历史流转记录

审核通过后不要求自动发布，但必须进入可发布状态。

### 9.5 驳回

驳回时必须填写驳回原因。

首期要求：

- 驳回原因不能为空
- 驳回后内容状态进入 `rejected`
- 编辑可查看驳回原因和驳回时间
- 驳回记录必须保留在历史记录中

### 9.6 发布

发布是内容进入门户可见状态的关键动作。

首期要求：

- 仅 `review` 审核通过的内容可发布
- 发布动作必须记录发布时间、发布人、发布任务号
- 发布后状态进入 `published`
- 发布后触发详情页、栏目页、首页聚合位、搜索索引的影响计算

### 9.7 下线

下线用于终止已发布内容的门户可见性。

首期要求：

- 仅 `published` 内容可下线
- 下线必须记录原因（可选增强，但建议首期支持）
- 下线后状态进入 `offline`
- 下线后触发门户详情页下线及相关聚合页重建

### 9.8 历史记录与审计

生命周期模块必须提供状态流转历史。

首期要求：

- 记录每次状态变化前后值
- 记录操作人、时间、操作说明
- 记录驳回原因、下线原因等附加信息
- 支持在内容详情页查看流转历史

### 9.9 发布前校验

正式发布前应统一校验内容是否具备发布条件。

首期至少校验：

- 内容状态正确
- 栏目有效
- 模板有效
- 媒体引用有效
- 必填字段完整
- SEO 基础字段满足最低要求（可只做弱校验提示）

## 10. 数据模型建议

建议在内容对象中保留以下生命周期相关字段：

| 字段 | 说明 | 是否必填 |
| --- | --- | --- |
| id | 内容唯一标识 | 是 |
| status | 当前生命周期状态 | 是 |
| reviewStatus | 审核结果扩展字段（可选） | 否 |
| submitterId | 送审人 | 否 |
| submittedAt | 送审时间 | 否 |
| reviewerId | 最近审核人 | 否 |
| reviewedAt | 最近审核时间 | 否 |
| rejectionReason | 最近驳回原因 | 否 |
| publisherId | 发布人 | 否 |
| publishedAt | 发布时间 | 否 |
| offlineOperatorId | 下线操作人 | 否 |
| offlineAt | 下线时间 | 否 |
| offlineReason | 下线原因 | 否 |
| versionNo | 当前内容版本号 | 是 |
| lastStateChangedAt | 最近状态变更时间 | 是 |

建议独立状态流转记录表至少包含：

- historyId
- contentId
- fromStatus
- toStatus
- operatorId
- operatorRole
- reason
- createdAt
- taskId（如与发布任务相关）

## 11. 业务规则

### 11.1 编辑规则

- `draft`、`rejected` 状态允许编辑
- `review` 状态默认不允许直接编辑，需撤回或驳回后修改（首期可采用“驳回后修改”模式）
- `published` 内容若需修改，首期建议生成新编辑版本或先下线后修订，具体实现可在后续细化

### 11.2 审核规则

- 审核人不能是内容当前提交人（建议规则）
- 审核意见必须留痕
- 驳回必须填写驳回原因
- 审核通过不等于自动静态化产物已发布完成，仍需走发布中心流程

### 11.3 发布规则

- 内容必须具备完整发布条件才允许发布
- 发布成功后门户才可见
- 若发布任务失败，状态不应被错误标记为 `published`
- 发布状态与发布执行结果需要一致

### 11.4 下线规则

- 下线内容必须立即从门户可见集合移除
- 下线后历史记录保留，不允许当作从未发布处理
- 下线内容若重新上线，必须重新审核

### 11.5 审计规则

- 每次关键状态变化必须记录
- 历史记录不可被普通编辑篡改
- 审计日志与业务历史记录应可关联查询

## 12. 权限设计

建议生命周期相关权限编码如下：

| 权限编码 | 说明 |
| --- | --- |
| `content:item:view` | 查看内容 |
| `content:item:create` | 创建内容 |
| `content:item:update` | 编辑内容 |
| `content:item:submit-review` | 提交审核 |
| `content:item:review` | 审核内容 |
| `content:item:reject` | 驳回内容 |
| `content:item:publish` | 发布内容 |
| `content:item:offline` | 下线内容 |
| `content:item:history:view` | 查看流转历史 |
| `content:item:impact:view` | 查看发布影响范围 |

## 13. 页面结构建议

内容生命周期相关信息建议嵌入内容中心与审核中心。

首期页面建议至少包含：

- 内容列表状态标签
- 内容详情页状态卡片
- 送审按钮
- 驳回按钮
- 发布按钮
- 下线按钮
- 流转历史面板
- 驳回原因查看区
- 发布前校验提示区

## 14. 与发布中心的契约

内容生命周期模块必须向发布中心提供明确的触发条件和影响对象。

首期至少包括：

- `review -> published`：触发详情页、栏目页、首页聚合位、搜索索引重建
- `published -> offline`：触发详情页下线、栏目页聚合更新、首页推荐位移除、搜索索引更新
- 内容字段变更后再次发布：触发受影响详情页和相关聚合位更新

生命周期模块不负责实际执行门户生成，但负责保证“何时允许发布、何时允许下线”的规则稳定。

## 15. 接口建议

首期建议接口分组如下：

- `POST /api/contents`：创建内容草稿
- `PUT /api/contents/{id}`：更新草稿或被驳回内容
- `POST /api/contents/{id}/submit-review`：提交审核
- `POST /api/contents/{id}/approve`：审核通过
- `POST /api/contents/{id}/reject`：驳回内容
- `POST /api/contents/{id}/publish`：发布内容
- `POST /api/contents/{id}/offline`：下线内容
- `GET /api/contents/{id}/history`：查看流转历史
- `GET /api/contents/{id}/precheck`：查看发布前校验结果
- `GET /api/contents/{id}/impact`：查看发布影响范围

## 16. 验收标准

### 16.1 功能验收

- 内容可在 `draft / review / rejected / published / offline` 间按规则流转
- 送审、驳回、发布、下线都具备明确入口
- 能查看驳回原因和流转历史
- 能在发布前获得明确校验结果
- 下线后门户不再展示该内容

### 16.2 业务验收

- 编辑与审核职责边界清晰
- 门户只消费 `published` 状态内容
- 审核通过后的发布动作可正确触发发布中心
- 已下线内容重新上线必须重新审核

### 16.3 风险控制验收

- 非法状态跳转被阻止
- 驳回无原因被阻止
- 发布条件不满足时被阻止
- 发布失败不会错误标记内容状态
- 状态流转全过程可审计、可回溯

## 17. 后续扩展位

以下能力保留到后续阶段，不纳入本期生命周期交付：

- 多级审核流
- 定时发布
- 撤回审核
- 审核委托与转办
- 内容版本对比与差异查看
- 更复杂的工作流引擎配置

## 18. 当前结论

对于 L1 而言，内容生命周期不是内容中心的附属细节，而是连接编辑、审核、发布和门户展示的核心规则层。

在实施顺序上，内容生命周期必须与栏目管理、模板管理和发布契约同步定义；在产品交付上，它应当作为 L1 第三份详细 PRD 尽快定稿，以保证后续后台标准版和门户标准版都建立在统一状态机之上。
