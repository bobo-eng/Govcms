# GovCMS 角色体系定义稿

## 1. 文档信息

- 文档名称：角色体系定义稿
- 版本：V1.0
- 当前状态：待评审
- 最近更新：2026-03-17
- 适用范围：GovCMS 后台标准版、CMS 平台共享层、门户发布链路相关角色定义
- 关联文档：`docs/03-backoffice-prd.md`、`docs/09-content-lifecycle-prd.md`、`docs/15-publish-center-mvp-prd.md`

## 2. 背景与目标

当前项目已经形成内容生命周期、审核工作区、发布中心 MVP 的核心闭环，但“谁能看什么、谁能做什么、谁负责什么对象”的产品边界仍未正式定稿。

本文件的目标不是简单罗列已有代码中的角色编码，而是将 GovCMS 的角色体系提升为统一产品基线，用于约束：

- 菜单与页面可见范围
- 核心动作权限范围
- 典型业务流程参与边界
- 后续导航、专题、站点治理与发布中心的职责分工

## 3. 角色分层

GovCMS 角色体系按四层定义：

| 分层 | 角色 | 定位 |
| --- | --- | --- |
| 平台治理层 | 超级管理员 | 负责全局系统治理与跨站点管理 |
| 站点治理层 | 站点管理员 | 负责本站点组织、配置与发布治理 |
| 业务执行层 | 编辑 / 审核员 / 发布员 | 负责内容生产、审核与正式发布执行 |
| 只读层 | 访客 / 只读 | 负责查看授权页面与数据，不执行写操作 |

## 4. 角色定义

### 4.1 超级管理员

超级管理员是平台级例外角色，不受普通业务角色矩阵限制。

职责范围：

- 用户、角色、权限、菜单管理
- 全局站点配置与跨站点治理
- 全局模板、发布、审计与异常处理
- 作为平台兜底角色处理特殊问题

不建议承担日常业务内容生产工作。

### 4.2 站点管理员

站点管理员是本站点治理角色，不等于超级管理员。

职责范围：

- 本站点基础信息与配置
- 栏目、导航、模板、专题配置
- 本站点发布配置与发布总控
- 本站点发布任务查看、执行、回滚与异常处理

不负责：

- 全局用户/角色/权限管理
- 全局菜单管理
- 跨站点治理策略制定

### 4.3 编辑

编辑负责内容生产。

职责范围：

- 创建内容
- 编辑内容
- 保存草稿
- 提交审核
- 查看自身内容的状态、驳回意见与发布结果

不负责审核通过与正式发布。

### 4.4 审核员

审核员负责内容质量把关。

职责范围：

- 查看待审核内容
- 审核通过
- 驳回并填写驳回原因
- 查看审核历史与必要预览结果

不负责内容创建与正式发布。

### 4.5 发布员

发布员是正式发布动作角色。

职责范围：

- 发布前校验
- 影响范围确认
- 正式发布
- 下线
- 回滚
- 查看发布任务、产物与日志

发布员不天然等于站点管理员。两者可以被同一账号兼任，但产品定义上必须分开。

### 4.6 访客 / 只读

访客 / 只读只访问被授权页面与对象，不执行写操作。

职责范围：

- 查看内容、栏目、模板、站点等授权信息
- 查看报表、只读详情与只读任务信息

不承担任何业务流转动作。

当前仓库已有角色编码如下：

| 当前编码 | 产品定义 |
| --- | --- |
| `admin` | 超级管理员 |
| `editor` | 编辑 |
| `reviewer` | 审核员 |
| `publisher` | 发布员 |
| `viewer` | 访客 / 只读 |

本期说明：

- `站点管理员` 作为下一阶段新增正式角色，本轮先完成产品定义，不要求立即落到权限与菜单实现。
- 当前 `publisher` 不上升为 `站点管理员`，避免把动作角色和治理角色混淆。

## 10. 关键规则

- 超级管理员是例外角色，不受业务角色矩阵限制。
- 站点管理员不是超级管理员，不拥有全局用户/角色/权限/菜单管理权限。
- 发布员不天然等于站点管理员；两者可以兼任，但定义必须分离。
- 菜单可见性由角色权限矩阵推导，不再单独定义另一套例外规则。
- 页面入口与动作权限必须能回溯到角色矩阵，不允许页面私自放宽口径。

## 11. 评审问题

- 超级管理员与站点管理员是否彻底分层？
- 发布员是否只是动作角色，而不是站点治理角色？
- 编辑是否需要具备栏目/模板的有限读取能力？
- 审核员是否允许查看模板预览但不编辑模板？
- 站点管理员未来是否允许兼任编辑/审核/发布动作角色？

## 12. 验收标准

- 每个角色都能回答“能看什么、能做什么、不能做什么”。
- `超级管理员` / `站点管理员` / `发布员` 之间不存在职责重叠歧义。
- 后续任何页面、按钮、接口都能映射回角色矩阵。

## 13. 当前结论

GovCMS 的角色体系不应再以“代码中有哪些角色编码”为主导，而应以上述产品定义为准。

本稿定稿后，后续导航、专题、站点治理和发布中心增强，都必须以角色体系作为权限与菜单设计基线。
