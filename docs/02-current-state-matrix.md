# GovCMS 现状对照矩阵

## 1. 当前代码概览

当前仓库已经具备以下可运行代码：

- 后端控制器：认证、仪表盘、用户、角色、权限、菜单、内容、站点、栏目、模板、发布、媒体
- 前端页面：登录、仪表盘、用户、角色、权限、菜单、内容、审核工作区、发布中心、站点、栏目、模板、媒体
- 已有测试：仪表盘、站点、菜单权限、栏目、模板、生命周期、发布中心、渲染预览的控制器与服务测试

这说明当前仓库已经从“后台原型”推进到“后台基座 + 栏目/模板 + 生命周期 + 发布中心 MVP”的阶段，但距离完整门户标准版和正式交付仍有明显差距。

## 3. 技术现状与目标差距

### 当前技术现状

- 后端：`Spring Boot 3.2 + Spring Security + Spring Data JPA + JWT`
- 前端：`Vue 3 + TypeScript + Vite + Ant Design Vue`
- 数据库：`MySQL 8`
- 部署形态：本地 `spring-boot:run` 与前端 `npm run dev`
- 文件存储：本地文件系统 `./storage/media` 与 `./storage/publish`

### 正式目标技术口径

- 数据库目标：`KingbaseES`
- 应用服务器目标：`TongWeb`
- Web 层目标：`Nginx`
- 运行环境目标：国产 OS / 政务云兼容
- 安全目标：国密传输、国密签名、国密数据保护、审计与回滚能力

## 4. 当前代码可直接复用部分

- 认证与权限框架
- 后台前端工程基础设施
- 用户、角色、权限、菜单管理能力
- 站点、栏目、模板、媒体的基础数据结构与页面骨架
- 内容生命周期与审核/发布最小链路
- `RenderContextAssembler` / `PortalRenderService` 提供的受控渲染基础
- 发布任务、产物、回滚记录的 MVP 实现基础

## 5. 当前代码必须升级或重构部分

- 角色体系需要从“代码角色”升级为“正式产品角色体系”
- 站点模型需要升级为完整站点治理模型，并引入站点管理员
- 需要补齐独立导航对象与导航管理能力
- 需要补齐独立专题对象与专题管理能力
- 发布链路需要从“同步执行 MVP”升级为“生产级发布编排”
- 安全能力需要从普通 JWT 升级到满足国密要求的正式方案
- 部署能力需要从本地开发口径升级为正式信创部署口径

## 6. 当前阶段结论

当前项目已经形成可演示的“内容生命周期 + 审核 + 发布中心 MVP”主链路。接下来的重点不应是盲目扩展门户页面，而应先完成角色体系定稿，再完成导航 / 专题 / 站点管理员建模，之后再进入下一阶段实现。
