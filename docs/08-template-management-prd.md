# GovCMS 模板管理 PRD

## 1. 文档信息

- 文档名称：模板管理 PRD
- 所属阶段：`L1 CMS 平台共享层` / `L2 后台标准版` / `L3 门户标准版`
- 当前状态：V1.0 初稿
- 最近更新：2026-03-11
- 适用范围：CMS 建站、门户页面渲染、静态化发布、预览与模板版本管理

## 2. 产品定位

模板管理是 GovCMS 中连接“后台配置”和“门户页面生成”的核心模块，用于把站点、栏目、内容、专题等结构化数据稳定渲染成门户页面。

模板不是简单的前端页面文件，而是标准化的页面渲染规则集合，承担以下职责：

- 定义页面结构
- 定义页面区块与数据位
- 定义不同页面类型的渲染契约
- 约束站点、栏目、内容与专题的展示方式
- 为门户静态化发布提供可执行输入

在 GovCMS 的产品体系里，模板管理必须服务于“CMS 驱动建站 + 门户静态化发布”的总体模式，因此首期模板管理强调标准化、稳定性、可追溯和可发布，而不是强调自由拖拽和低代码搭建。

## 3. 产品目标

### 3.1 目标

- 建立统一的门户页面模板体系
- 支持站点、栏目、内容详情、专题等对象进行模板绑定
- 支持模板编辑、预览、版本管理和生效控制
- 为发布中心提供稳定的渲染契约和变更影响识别能力
- 为门户静态化生成提供标准化输入

### 3.2 非目标

首期模板管理不负责以下能力：

- 低代码拖拽式可视化搭建器
- 复杂组件市场和组件交易机制
- 多端自适应模板编排引擎
- 页面级 AB 测试与个性化推荐策略
- 用户自主安装第三方模板插件体系

## 4. 适用角色

| 角色 | 使用目标 |
| --- | --- |
| 平台管理员 | 管理模板规范、状态、权限与发布安全边界 |
| 站点管理员 | 为站点、栏目、专题配置模板并控制生效范围 |
| 编辑 | 预览内容在不同模板下的呈现效果 |
| 审核人员 | 核对模板预览结果是否符合发布要求 |
| 运维人员 | 确认模板变更影响范围、配合执行模板发布与回滚 |

## 5. 业务边界

模板管理与其他模块的关系如下：

- 与 `Site`：站点需要绑定首页模板、默认 404 模板、公共布局模板规则
- 与 `Column`：栏目必须绑定栏目页模板，可选择默认详情模板规则
- 与 `Content`：内容详情页渲染依赖内容模板或栏目继承的详情模板
- 与 `TopicPage`：专题页需要绑定专题模板
- 与 `Navigation`：导航展示受公共布局与页面模板影响
- 与 `PublishJob`：模板变更会影响受绑定对象对应的页面重建范围
- 与 `Portal`：门户页面最终由模板和已发布数据共同生成

## 6. 模板体系设计

### 6.1 模板类型

首期模板体系固定为以下 5 类：

- 首页模板
- 栏目页模板
- 内容详情页模板
- 专题页模板
- 404 页面模板

首期不支持任意自定义页面类型。

### 6.2 模板层级

首期建议支持以下 3 层绑定关系：

- 站点级默认模板
- 栏目级模板
- 专题级模板

内容详情模板优先级建议如下：

1. 内容明确指定模板
2. 所属栏目默认详情模板
3. 站点默认详情模板

### 6.3 模板结构模型

模板首期采用“标准页面模板 + 结构化区块配置”模式，不采用自由拖拽模式。

模板结构建议至少包含：

- 页面基础布局
- 头部区块
- 导航区块
- 面包屑区块
- 主内容区块
- 侧栏区块（可选）
- 推荐位区块（可选）
- 页脚区块
- SEO 元信息区块

## 7. 核心场景

### 7.1 新站初始化

站点管理员新建站点后，为首页、栏目页、详情页和 404 页面配置默认模板，以便后续栏目和内容可直接生成门户页面。

### 7.2 栏目绑定模板

站点管理员为不同栏目配置不同栏目页模板，满足新闻类、通知类、专题聚合类等不同展示需求。

### 7.3 内容预览

编辑在发布内容前，需要查看内容在当前详情模板下的真实呈现效果，确认版式、字段和媒体展示正常。

### 7.4 模板升级

运营过程中需要优化门户视觉和布局，模板管理员通过新版本模板替换现有模板，并评估对首页、栏目页、详情页和专题页的影响。

### 7.5 发布与回滚

模板变更后，发布中心根据模板绑定范围识别受影响页面；若模板发布异常，需要回滚到上一稳定模板版本。

## 8. 核心设计原则

- 模板必须标准化，优先保证门户输出稳定，而不是强调编辑自由度
- 模板必须与页面类型严格绑定，避免模板语义混乱
- 模板变更必须可追溯、可预览、可回滚
- 模板必须为发布中心提供清晰影响范围
- 模板管理必须与栏目、内容、专题管理解耦，但绑定关系必须清晰可见

## 9. 功能范围

### 9.1 模板列表与检索

支持按站点、模板类型、状态、使用范围查看模板列表。

功能要求：

- 支持按站点筛选模板
- 支持按模板类型筛选
- 支持按状态筛选
- 支持按模板名称、编码搜索
- 支持查看模板当前绑定数量和最近更新时间

### 9.2 新增模板

支持创建标准模板。

新增时需配置：

- 所属站点（平台级公共模板可扩展）
- 模板名称
- 模板编码
- 模板类型
- 模板描述
- 模板布局结构
- 模板区块配置
- SEO 元信息规则
- 状态
- 初始版本说明

### 9.3 编辑模板

支持修改模板结构与展示规则。

编辑时重点关注：

- 是否改变页面结构
- 是否影响既有数据位映射
- 是否影响导航、面包屑、推荐位等公共区块
- 是否影响静态发布产物路径或依赖资源

### 9.4 模板绑定管理

支持将模板绑定到站点、栏目、详情页规则或专题。

首期要求：

- 首页模板按站点绑定
- 栏目页模板按栏目绑定
- 详情页模板支持站点默认和栏目默认
- 专题模板按专题绑定
- 404 模板按站点绑定

### 9.5 模板预览

模板必须支持预览。

首期要求：

- 支持空模板结构预览
- 支持结合样例数据预览
- 支持结合真实栏目或真实内容数据预览
- 支持预览 PC 门户输出结果
- 预览不等同于发布，不生成正式产物

### 9.6 模板版本管理

模板需要支持版本管理。

首期要求：

- 每次保存可形成版本记录
- 版本包含版本号、修改人、修改时间、版本说明
- 支持查看历史版本
- 支持回滚到历史稳定版本
- 回滚操作需二次确认

### 9.7 模板状态管理

模板首期支持以下状态：

- `draft`：草稿
- `active`：启用
- `disabled`：停用

状态要求：

- 草稿模板可编辑、可预览，但不可用于正式发布
- 启用模板可绑定并进入发布流程
- 停用模板不可继续绑定到新对象

### 9.8 模板依赖与影响分析

模板管理需要展示模板影响范围。

首期要求：

- 显示模板绑定了哪些站点、栏目、专题
- 显示模板变更会影响多少页面类型
- 模板变更前提示影响范围
- 模板发布前可查看预估重建页面数量

### 9.9 模板安全约束

首期模板体系必须限制不安全能力。

首期要求：

- 不允许在模板中执行任意服务端脚本
- 不允许直接访问未发布数据
- 不允许引用未受控的外部资源地址
- 不允许绕过字段白名单直接拼接任意数据结构

## 10. 数据模型建议

建议核心字段如下：

| 字段 | 说明 | 是否必填 |
| --- | --- | --- |
| id | 模板唯一标识 | 是 |
| siteId | 所属站点 ID | 是 |
| name | 模板名称 | 是 |
| code | 模板编码，站点内唯一 | 是 |
| type | 模板类型 | 是 |
| description | 模板说明 | 否 |
| layoutSchema | 页面布局结构定义 | 是 |
| blockSchema | 区块与数据位定义 | 是 |
| seoSchema | SEO 配置规则 | 否 |
| status | 模板状态 | 是 |
| version | 当前版本号 | 是 |
| activeVersionId | 当前生效版本 ID | 否 |
| previewDataSource | 默认预览数据源类型 | 否 |
| dependencyCount | 当前绑定对象数量 | 否 |
| createdBy | 创建人 | 是 |
| updatedBy | 更新人 | 是 |
| createdAt | 创建时间 | 是 |
| updatedAt | 更新时间 | 是 |

建议模板版本表至少包含：

- versionId
- templateId
- versionNo
- layoutSnapshot
- blockSnapshot
- seoSnapshot
- changeLog
- createdBy
- createdAt

## 11. 业务规则

### 11.1 唯一性规则

- `code` 在同一站点内唯一
- 同一站点内，同一类型允许多个模板存在，但同一业务对象同一时刻只能生效一个模板绑定

### 11.2 类型绑定规则

- 首页模板只能绑定站点首页
- 栏目页模板只能绑定栏目
- 详情页模板只能绑定内容详情规则
- 专题模板只能绑定专题页
- 404 模板只能绑定站点 404 页面

### 11.3 生效规则

- 仅 `active` 状态模板可参与正式发布
- 草稿模板不可进入发布链路
- 停用模板不可新绑定，但已绑定模板需先解绑或替换后再停用

### 11.4 预览规则

- 预览必须使用受控数据源
- 预览结果只用于校验视觉和结构，不代表正式发布结果完全一致
- 模板预览时需展示模板版本号和预览数据来源

### 11.5 回滚规则

- 仅可回滚到已有历史版本
- 回滚后需形成新的回滚记录，而不是覆盖历史记录
- 回滚后需重新进入发布流程，不自动对外生效

### 11.6 发布规则

- 模板变更必须输出受影响对象范围
- 模板变更需由发布中心统一纳入整站或增量发布
- 模板未通过预览校验时，不建议进入正式发布

## 12. 权限设计

建议模板权限编码如下：

| 权限编码 | 说明 |
| --- | --- |
| `portal:template:view` | 查看模板 |
| `portal:template:create` | 新增模板 |
| `portal:template:update` | 编辑模板 |
| `portal:template:delete` | 删除模板 |
| `portal:template:bind` | 绑定模板 |
| `portal:template:preview` | 预览模板 |
| `portal:template:version:view` | 查看模板版本 |
| `portal:template:rollback` | 回滚模板版本 |
| `portal:template:impact:view` | 查看模板影响范围 |

## 13. 页面结构建议

模板管理页面建议采用“三段式结构”：

- 左侧：模板列表、筛选和搜索
- 中间：模板基础信息、区块配置、绑定关系
- 右侧：预览区 / 影响范围 / 版本记录

页面至少包含以下操作入口：

- 新增模板
- 编辑模板
- 复制模板（可作为后续增强）
- 绑定模板
- 解除绑定
- 预览模板
- 查看版本
- 回滚版本
- 查看影响范围
- 停用模板

## 14. 与发布中心的契约

模板管理模块必须向发布中心输出标准影响类型，首期至少包括：

- 首页重建
- 栏目页重建
- 详情页重建
- 专题页重建
- 404 页面重建
- 公共导航与布局重建
- 静态资源引用刷新

模板管理模块不直接执行正式发布，但必须提供稳定的渲染契约、绑定结果和影响范围。

## 15. 接口建议

首期建议接口分组如下：

- `GET /api/templates`：读取模板列表
- `GET /api/templates/{id}`：读取模板详情
- `POST /api/templates`：创建模板
- `PUT /api/templates/{id}`：更新模板
- `PUT /api/templates/{id}/status`：切换模板状态
- `POST /api/templates/{id}/preview`：模板预览
- `GET /api/templates/{id}/versions`：读取模板版本列表
- `POST /api/templates/{id}/rollback`：回滚模板版本
- `GET /api/templates/{id}/impact`：查看模板影响范围
- `POST /api/templates/{id}/bindings`：绑定模板到对象
- `DELETE /api/templates/{id}/bindings/{bindingId}`：解除模板绑定

## 16. 验收标准

### 16.1 功能验收

- 能创建不同类型的标准模板
- 能编辑模板结构和区块配置
- 能按站点、栏目、专题完成模板绑定
- 能查看模板版本并执行回滚
- 能基于样例数据或真实数据完成预览
- 能查看模板影响范围

### 16.2 业务验收

- 首页、栏目页、详情页、专题页和 404 页面都存在明确模板来源
- 栏目和内容可按绑定规则找到最终使用模板
- 模板变更后可被发布中心正确识别受影响页面
- 草稿模板不会误进入正式发布

### 16.3 风险控制验收

- 错误类型模板无法绑定到不匹配对象
- 模板停用前需处理已绑定关系
- 模板回滚全过程可追溯
- 预览与正式发布的数据边界清晰
- 模板中不存在未受控脚本执行能力

## 17. 后续扩展位

以下能力保留到后续阶段，不纳入本期模板管理交付：

- 可视化拖拽式搭建器
- 模板组件市场
- 多终端模板自适配编排
- 模板导入导出市场化分发
- 多语言模板渲染策略
- 更复杂的区块级权限与协作能力

## 18. 当前结论

对于 L1 而言，模板管理不是门户阶段的附属能力，而是共享平台层必须提前定义的核心能力。

在实施顺序上，模板管理需要与栏目管理、内容生命周期和发布契约同步设计；在产品交付上，它应当作为 L1 第二份详细 PRD 尽快定稿，以避免后续后台和门户实现反复返工。
